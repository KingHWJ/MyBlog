# MyBlog 技术方案

## 1. 站点定位

MyBlog 是一个面向后端与大数据学习者的中文知识库技术站，首版强调：

- 知识入口清晰
- 专题结构稳定
- 内容可持续扩展
- 发布与维护成本可控

## 2. 内容结构

站点采用“专题文档 + 更新文章”的双轨结构：

- `roadmap/`：学习路线与方法论
- `backend/`：后端基础主线
- `bigdata/`：大数据专题
- `updates/`：更新归档与独立文章

其中：

- 专题文档用于长期维护和持续修订
- 更新文章用于记录新的输出，再按需要沉淀回专题

## 3. Frontmatter 规范

所有内容页统一使用以下 frontmatter 字段：

- `title`：页面标题
- `description`：页面简介
- `date`：发布日期，格式建议为 `YYYY-MM-DD`
- `tags`：标签数组
- `category`：内容分类
- `featured`：是否出现在首页推荐
- `draft`：是否为草稿

## 4. 导航与侧边栏

顶部导航固定为：

- 首页
- 学习路线
- 后端基础
- 大数据
- 归档/更新

侧边栏采用手工配置，确保目录层次稳定，首版不引入自动生成规则。

当前首版目录约定如下：

- `docs/index.md`：首页
- `docs/roadmap/`：学习路线
- `docs/backend/`：后端专题
- `docs/bigdata/`：大数据专题
- `docs/updates/`：更新归档
- `docs/.vitepress/theme/`：主题扩展组件与样式

首页“最近更新”使用 VitePress 数据加载器生成，避免手工维护首页卡片列表。

## 5. 工程与发布

- 包管理器：`pnpm`
- 站点框架：`VitePress`
- 测试：`Vitest`
- 部署：`GitHub Pages`

发布流程：

1. 在开发分支上完成修改
2. 本地运行测试与构建验证
3. 合并时使用 `--no-ff`
4. 由 GitHub Actions 负责构建并发布到 GitHub Pages

### 5.1 开发命令

```bash
pnpm install
pnpm docs:dev
pnpm test
pnpm docs:build
pnpm worktree:path codex/your-branch
```

补充约定：

- `docs:dev` 的 VitePress 开发服务在 `docs/.vitepress/config.ts` 中开启了轮询监听
- 这样做是为了让 `git worktree` 目录下的 Markdown 文件改动也能稳定触发热更新
- 该设置只作用于本地开发服务器，不影响 `docs:build`

### 5.2 Node 版本约定

仓库以 Node.js 22 LTS 为默认版本约定，并通过 `.nvmrc` 固定版本。

如果本地使用更高版本的 Node.js，`pnpm` 可能会给出 engine 警告，但 GitHub Actions 会以 Node.js 22 作为标准执行环境。

### 5.3 Worktree 目录约定

为了和 Codex 个性化规则保持一致，仓库的 worktree 不再使用项目内 `.worktrees/` 目录，而是统一创建在：

```text
~/Codes/worktree/<项目相对目录>/<项目名>/<分支层级>
```

当前项目 `/Users/hwj/Codes/MyProjects/MyBlog` 的示例：

```bash
pnpm worktree:path codex/example-task
# /Users/hwj/Codes/worktree/MyProjects/MyBlog/codex/example-task
```

推荐创建方式：

```bash
BRANCH="codex/example-task"
TARGET="$(pnpm --silent worktree:path "$BRANCH")"
git worktree add "$TARGET" -b "$BRANCH" main
```

如果历史上已经存在直接占用 `~/Codes/worktree/<项目相对目录>/<项目名>` 的旧 worktree，需要先完成迁移或清理，再切换到“项目目录 / 分支子目录”的新结构。

需要新建 worktree 时，应先确认是否真的需要创建独立工作区，再执行创建，不直接默认新建。

## 6. 开发约束

- 不在主分支直接开发
- Git 提交按小模块拆分，并使用中文提交信息
- 代码和配置中的注释优先使用中文
- 开发过程中同步维护技术文档，避免实现与文档脱节

更细的日常维护步骤，见 [操作手册](./操作手册.md)。
